SEASONAL WEATHER PREDICTION DASHBOARD: EXPLANATION GUIDE
=====================================================

This guide highlights the key technical and academic points you should emphasize when explaining this project.

1. SYSTEM ARCHITECTURE & TECH STACK
------------------------------------
* The "Why": The project avoids heavy frameworks (like Flask/FastAPI) to ensure zero-latency and high reliability for research data.
* The Stack: 
    - Frontend: Vanilla HTML5, CSS3 (Glassmorphism), and JavaScript.
    - Backend: Python-based ML Pipeline using XGBoost and Scikit-Learn (Neural Network).
    - API Bridge: A lightweight threaded HTTP server (server.py) connects the browser to the Python ML engine via stdin/stdout pipes.
    - Data Flow: The pipeline fetches data -> processes it -> saves as CSV -> JavaScript reads via PapaParse.
    - Live Inference: Dashboard sends prediction requests to /api/predict_batch -> server.py spawns predict_live.py -> models loaded from .joblib -> JSON response returned.

2. DATA STRATEGY (OPEN-METEO API)
----------------------------------
* Timeline: 24 years of historical data (2001-2024) for 10 UK regions (London, Manchester, Middlesbrough, etc.).
* Feature Engineering (Critical): This is the "brain" of the project.
    - Lags (1-day & 7-day): Allows the model to "remember" yesterday's weather.
    - Rolling Means (7-day & 30-day): Smooths out noise to identify seasonal trends.
    - Temporal Firewall: All lag features use shift(1) BEFORE calculation, mathematically guaranteeing zero data leakage.

3. THE MACHINE LEARNING MODELS
-------------------------------
* XGBoost (The Winner): A "Gradient Boosted Decision Tree" model. It handles non-linear weather patterns better than simple models.
* Neural Network (MLP): A Multi-Layer Perceptron (128, 64, 32 architecture) used as a secondary "deep learning" baseline.
* Random Forest: Used as the initial baseline model (eliminated during optimization).
* Validation: Walk-Forward Validation (2001-2021 for training, 2022-2024 for testing).
* Live Retraining: An autonomous daemon (live_pipeline.py) runs a catch-up engine at midnight, retraining models on each newly completed day's data.

4. ADVANCED "ACADEMIC" FEATURES
-------------------------------
* Explainable AI (SHAP): Solves the "Black Box" problem by visually showing which features drive the AI's prediction.
* Anomaly Detection (Z-Score): Automatically identifies "Extreme Weather Events" (Heatwaves/Cold Snaps) using per-region statistical deviations.
* Concept Drift Analytics: Monitors how model accuracy drifts over time as the climate changes. The drift_history.csv tracks daily XGBoost and Neural Network MAE.
* 14-Day Scientific Validation: Tab 5 compares live ML predictions against ECMWF and GFS baselines from the Open-Meteo Forecast API.

5. LIVE PIPELINE & AUTONOMOUS RETRAINING
-----------------------------------------
* Midnight Catch-Up Engine: Every day at midnight, the system fetches the previous day's completed weather data and retrains both models.
* The "Yesterday" Rule: The system strictly defines "Finished Day" as Today - 1 Day, preventing API errors from requesting data that doesn't exist yet.
* Manual Retrain: A "Neural Reset" button in Tab 4 triggers force_retrain_hourly.py, which evaluates model drift on completed days only.
* Signal-Flag Mechanism: The dashboard writes a retrain.flag file; the background daemon picks it up non-blockingly.

6. PERFORMANCE ENGINEERING
---------------------------
* Dual-Read Strategy: predict_live.py uses a byte-seek tail-read for live requests (last 30 days) and a full CSV read for deep historical queries.
* Memory Management: float32 downcasting reduces RAM footprint by 50% during pipeline execution.
* Pre-Calculated KPIs: Heavy metrics (total records, regional counts) are computed on the backend and served as a lightweight JSON file, preventing browser crashes on the 587MB dataset.

7. SIMULATION PURGE (ACADEMIC INTEGRITY)
-----------------------------------------
* CRITICAL: All Math.random() and np.random() simulation fallbacks have been permanently deleted from the codebase.
* If the ML API is offline, the dashboard displays "API Err" or "-- °C" instead of faking results.
* This ensures every number displayed on the dashboard is derived from authentic model inference weights.

8. UI/UX DESIGN (THE WOW FACTOR)
--------------------------------
* Glassmorphism: The "frosted glass" look provides a premium, modern feel.
* Interactive Predictor: Allows users to query the AI for specific dates (2001 to +365 days) and see the error margin.
* Thermal Color Logic: Sun icons dynamically change color based on predicted temperature (blue for cold, gold for mild, red for hot).
* Accessibility: Includes dark/light mode toggle, keyboard shortcuts (M for theme, B for tab cycle, Escape for modals), and responsive layouts.

TIPS FOR PRESENTATION:
----------------------
When presenting, emphasize:
1. Feature Engineering and the Temporal Firewall (shift-before-calculate) — this proves zero data leakage.
2. SHAP values — these show you understand not just "how" to build a model, but "why" the model behaves the way it does.
3. The Live Retraining Loop — this demonstrates production-grade MLOps, not just a static notebook.
4. The Simulation Purge — this proves academic honesty and scientific integrity.
